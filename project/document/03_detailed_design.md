## 2. フロントエンド詳細設計（PHP）
### 2.1 画面仕様
- 入力項目1: フォルダパス（必須）
- 入力項目2: メールアドレス（必須）
- ボタン: 決定
- 入力エラー画面: フォルダパス/メール不正時に不正内容と再入力導線を表示

### 2.1.1 画面表現責務（PHP / HTML / CSS / JavaScript）
- PHP責務: 画面状態（初期/確認前/エラー時/結果時）に応じたHTMLテンプレートを生成する。
- HTML責務: 入力フォーム、確認ダイアログ、エラー表示領域、結果表示領域のDOM構造を定義する。
- CSS責務: 可読性を損なわないレイアウト（入力欄、ボタン、エラーメッセージ、結果領域）を提供する。
- JavaScript責務: 確認ダイアログ制御、キャンセル時入力保持、送信前最終確認を制御する。
- 責務境界: 入力妥当性の最終判定はバックエンドが行い、フロントは表示と操作導線を担う。

### 2.2 確認ダイアログ仕様
- 表示内容: 入力したフォルダパス、メールアドレス
- 補足: フォルダパスに空白を含む場合も入力文字列を欠損なくそのまま表示する
- ボタン: OK / キャンセル
- OK: サーバへ要求送信
- キャンセル: 入力保持したまま元画面へ復帰

### 2.3 バリデーション仕様
- 空入力チェック
- メール形式チェック
- パス存在チェック（サーバ側で最終判定）
- Windows環境では、パスに空白を含むこと自体を不正扱いしない

### 2.4 文字コード受理仕様
- ブラウザ入力値（フォルダパス、メール）は UTF-8 / Shift-JIS（Windows-31J/CP932）を受け付ける。
- submit.php で受信した入力値は、`UTF-8`、`SJIS-win`、`CP932` の順で判定し、UTF-8へ正規化してバックエンドへ渡す。
- 判定不能な場合は実行要求を送らず、標準エラー応答形式に準じた画面エラーを表示する。

### 2.5 画面イベント仕様
| イベント | 条件 | 動作 | 結果 |
|---|---|---|---|
| 初期表示 | 画面アクセス | フォーム表示 | 入力待ち |
| 決定押下 | 入力不正 | 入力エラー画面表示（不正内容明示） | 入力再試行可能 |
| 決定押下 | 入力妥当 | 確認ダイアログ表示 | OK/キャンセル待ち |
| ダイアログOK | 確認済み | 実行要求送信（空白を含むパスは単一文字列として送信） | バックエンド実行開始 |
| ダイアログキャンセル | 任意 | ダイアログ閉じる | 入力値保持 |
| 実行失敗 | BE異常応答受信 | エラー表示領域へ整形表示 | 入力再試行可能 |

### 2.7 フロントエンドモジュール仕様
| モジュール | 主責務 | 入力 | 出力 |
|---|---|---|---|
| index.php | 初期画面HTML生成、入力フォーム描画、確認ダイアログ起動 | なし（初期アクセス） | 入力画面HTML |
| submit.php | 入力受信、文字コード正規化、バリデーション、バックエンド呼出し（空白を含むパスの単一文字列維持） | フォルダパス、メールアドレス | 結果画面HTML（成功/失敗） |
| frontend script（JavaScript） | ダイアログOK/キャンセル制御、入力保持 | 入力フォーム状態 | 実行要求送信または入力画面復帰 |
| frontend style（CSS） | 画面視認性の担保（入力/エラー/結果領域） | HTML構造 | スタイル適用済み画面 |

### 2.6 画面遷移仕様
| 画面/状態 | 遷移条件 | 遷移先 | 備考 |
|---|---|---|---|
| 入力フォーム | 決定押下かつ入力妥当 | 確認ダイアログ | 入力値をダイアログへ表示 |
| 入力フォーム | 決定押下かつ入力不正 | 入力エラー画面 | 不正内容を明示して再入力導線を表示 |
| 確認ダイアログ | OK押下 | 結果表示 | BEへ実行要求を送信 |
| 確認ダイアログ | キャンセル押下 | 入力フォーム | 入力値を保持 |
| 結果表示 | 入力画面へ戻る押下 | 入力フォーム | 再実行可能 |

## 3. バックエンド詳細設計（Python）
### 3.1 処理フロー
1. 要求受信
	- 入力パスは空白を含む場合も単一文字列として受理する
2. 入力文字列を UTF-8 / Shift-JIS 判定後に UTF-8 へ正規化
3. 指定パスからiファイル探索
4. iファイル読込時に UTF-8 / Shift-JIS 判定後に UTF-8 へ正規化
5. 関数情報抽出
6. 正規化（コメント/改行除去）
7. 社内サーバ情報との比較
8. result.csv生成（ユーザ指定パスと同一階層へ配置）
9. 一致ファイル集約コピー（ユーザ指定パスと同一階層へ配置、同名衝突時は連番付き新規フォルダを作成して上書き回避）
10. 完了メール送信
11. 異常発生時は標準エラー応答を生成して返却

### 3.2 モジュール設計
- input_validator.py: 入力検証
- encoding_normalizer.py: 文字コード判定・UTF-8正規化
- i_file_scanner.py: iファイル探索
- function_extractor.py: 関数抽出
- normalizer.py: コメント/改行除去
- matcher.py: 一致判定
- exporter.py: CSV/コピー出力
- notifier.py: メール通知
- db_updater.py: database.json日次更新
- error_formatter.py: 例外を標準エラー応答へ変換

### 3.3 主要関数I/F（例）
| モジュール | 関数 | 入力 | 出力 |
|---|---|---|---|
| encoding_normalizer.py | normalize_text_encoding | raw_text_or_bytes | utf8_text |
| i_file_scanner.py | scan_i_files | base_path | iファイル一覧 |
| function_extractor.py | extract_functions | i_file_path | 関数定義リスト |
| normalizer.py | normalize_logic | function_text | 正規化後文字列 |
| matcher.py | match_functions | user_functions, database | 一致結果 |
| exporter.py | export_results | match_result, out_dir | result.csv, 集約先（同名衝突時は連番付き） |
| notifier.py | send_complete_mail | email, summary | 送信成否 |
| error_formatter.py | format_error | exception, run_id | error_code, message, run_id |

### 3.4 ファイル仕様
#### database.json
- 用途: 社内サーバ情報・iファイル情報の保管
- 更新: 日次
- 格納例: 関数名、正規化ロジック、評価結果フォルダパス、更新日時

#### temporary_*.json
- 用途: 実行時の抽出結果保管
- 命名: 時刻+乱数
- 例: temporary_20260224_153000_843219.json

#### result.csv
- 用途: 一致結果一覧
- 主な列: 関数名, 一致可否, 対象iファイル, 評価結果フォルダ, 出力先, 備考
- 配置先: ユーザ指定パスと同一階層

#### copied_resultsフォルダ
- 用途: 一致結果フォルダ内ファイルの集約コピー
- 配置先: ユーザ指定パスと同一階層
- 命名規則: `copied_results` を第一候補とし、同名が存在する場合は `copied_results_1`, `copied_results_2` ... の順で未使用名を採番する
- 上書き方針: 既存フォルダは上書きしない

#### error_response.json（論理仕様）
- 用途: 異常時の標準応答
- 主な項目: error_code, message, run_id, detail(optional)

#### 応答JSONサンプル（実行API）
- 成功時:
	- `{"status":"ok","run_id":"20260224_033815_548041","input_path":"../test/sample_input","matched_count":1,"total_count":2,"result_csv":".../result.csv","copied_dir":".../copied_results"}`
- 失敗時:
	- `{"status":"error","error_code":"INVALID_EMAIL","message":"invalid email: invalid-mail","run_id":"20260224_033534_168372"}`

## 4. 例外処理設計
- 入力不正: エラー応答、処理中止
- パス未存在: エラー応答、処理中止
- iファイル未検出: エラー記録、処理中止
- 比較失敗: エラー記録、処理中止
- 通知失敗: リトライ方針またはエラー記録
- 想定外例外: イシュー登録情報を残し処理中止
- トレースバックはログに保持し、ユーザ応答では非表示とする

### 4.1 エラーコード案
| error_code | 条件 | 表示メッセージ（例） |
|---|---|---|
| INPUT_PATH_NOT_FOUND | 入力パス不存在 | 指定されたフォルダパスが存在しません。 |
| INVALID_EMAIL | メール形式不正 | メールアドレスの形式が正しくありません。 |
| INVALID_ENCODING | 文字コード判定不能 | 文字コードを判定できませんでした。UTF-8 または Shift-JIS を使用してください。 |
| I_FILE_NOT_FOUND | iファイル未検出 | 指定フォルダに i ファイルが見つかりません。 |
| INVALID_INPUT | その他入力不正 | 入力内容を確認してください。 |
| INTERNAL_ERROR | 想定外例外 | 処理中にエラーが発生しました。 |

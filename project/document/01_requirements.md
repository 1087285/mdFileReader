# 01 要件定義

## 1. 文書情報
- 文書名: 要件定義書
- プロジェクト名: topDownTest
- 作成日: 2026-03-02
- 作成者: GitHub Copilot
- 版数: v0.9

## 2. 目的
機能安全業務における単体評価テスト結果の紐づけ手段を定義する。

## 3. スコープ
### 3.1 対象
- C言語による組み込み系ソフトの単体評価結果再利用判定
- ユーザ指定パス上の i ファイルを入力とした関数単位の一致判定
- 一致結果の CSV 出力、関連ファイル集約、完了メール通知

### 3.2 非対象
- C ソース（.c）を直接入力とした比較
- ビルド/コンパイル処理そのもの
- 自動デプロイ

## 4. 前提条件
- 単体評価結果は社内サーバ上に存在し、関数単位でフォルダ管理される
- 社内サーバ側のフォルダ階層の深さは一定ではない
- 評価結果フォルダには html とプリプロセッサ処理後ソース（.i）が格納される
- コンパイルSW設定により .i ファイルのロジックが変化する
- 1つのソースに複数関数定義が存在する
- 結果再利用は「.i ロジック一致時のみ」許可する

## 5. 機能要件
### 5.1 入力要件
- ユーザ入力: フォルダパス、メールアドレス
- ユーザは機能安全対象ソフトを事前に .i へ変換し、アプリが参照可能なサーバへ配置する
- Web入力値（フォルダパス等）は UTF-8 と Shift-JIS（Windows-31J/CP932）の双方を受け付ける
- Windows環境で、フォルダパス中に空白を含む入力を受け付ける

### 5.2 処理要件
- iファイル探索
- 関数抽出
- コメント/改行除去による正規化
- iファイルおよび入力文字列の文字コード差異（UTF-8 / Shift-JIS）を吸収し、内部処理で UTF-8 に正規化する
- 関数単位の一致比較
- 抽出結果を一時JSONとして保存（複数同時実行を考慮した一意ファイル名）
- 社内サーバ情報を JSON で保持し、database.json を日次更新する

### 5.3 出力要件
- result.csv の生成
- 一致結果フォルダ内ファイルの集約コピー
- 集約コピー先フォルダと result.csv は、ユーザ指定パスと同一階層へ配置する
- 集約コピー先フォルダ名が既存と重複する場合は「_1」「_2」…の連番を付与して新規作成し、既存フォルダを上書きしない
- 完了メール送信
- ユーザ向け利用手順を HOWTOUSE.md に記載し、要求/出力変更時に更新確認を行う
- 異常時はトレースバックをそのまま返さず、ユーザ向け簡易メッセージと機械可読なエラー情報（JSON）を出力する

### 5.4 画面要件（フロントエンド）
- PHPでWebページを生成する
- PHPでHTMLを生成し、CSS/JavaScriptを用いて利用者にとって見やすい画面を構成する
- 入力項目は「フォルダパス」「メールアドレス」の2項目
- 決定ボタン押下後、確認ダイアログを表示する
- 確認ダイアログには入力値を表示し、OK/キャンセルを提供する
- OKでサーバ処理要求を送信し、キャンセル時は入力保持で元画面へ戻る
- フォルダパスまたはメールアドレスが不正な場合は、不正内容を明示した画面を表示する

## 6. 非機能要件
- 実行環境: Windows Server 2019 / XAMPP Apache
- デプロイ方式: 手動デプロイ
- 日次更新: database.jsonを毎日更新
- 開発方式: ウォーターフォール
- 開発基盤: GitHub Copilot を用いたクラウド開発
- 成果物管理: 工程ごとに md ファイルを作成する

## 6.1 運用要件
- database.json は日次で更新し、更新失敗時は運用担当へ通知・記録する
- 問題発生時はイシュー登録を行い、該当工程を特定して再実施する
- デプロイ先（Windows Server 2019 / XAMPP Apache）で手動デプロイ手順を維持する

## 6.2 保守要件
- 要求/要件/出力仕様が変更された場合、HOWTOUSE.md の更新要否を必ず確認し反映する
- エラーコード体系、メール送信方式、CSV列定義の確定後は本書を改版し履歴管理する
- 本書（01_requirements.md）の更新根拠は `agent/01_requirements_agent.md` の記載変更に限定し、他資料は参照情報として扱う
- 上記ルールの例外更新が必要な場合は、GitHub使用者のレビュー承認を必須とする

## 7. 例外・異常時要件
- 入力不正時は処理停止
- ファイル未存在時は処理停止
- 問題発生時はイシュー登録し停止
- 問題の織り込み工程を特定し、該当工程から再実施する
- 異常時の応答は「エラーコード・メッセージ・実行ID」を含む標準形式で返す
- Web画面では標準形式をもとに利用者が理解しやすい文面で表示する

## 8. 受入条件
### 8.1 正常系
- 指定パスのiファイルを処理し、result.csvと集約コピーを生成できる
- 集約コピー先フォルダと result.csv が、ユーザ指定パスと同一階層へ配置される
- 集約コピー先フォルダの同名衝突時に連番付き新規フォルダが作成され、既存フォルダが保持される
- 一致した関数のみ結果再利用対象として判定される
- 完了時に指定メールアドレスへ通知される
- UTF-8 および Shift-JIS（Windows-31J/CP932）入力で同等の処理結果を得られる

### 8.2 異常系
- 想定異常時に停止・記録ができる
- キャンセル操作時に入力値が保持される
- 問題発生時にイシュー登録フローへ移行できる
- パス不正・メール不正などで標準化されたエラー応答が返る

### 8.3 応答JSON受入基準
- 成功時は `status=ok` を返し、`run_id`、`matched_count`、`total_count`、`result_csv`、`copied_dir` を必須とする
- 失敗時は `status=error` を返し、`error_code`、`message`、`run_id` を必須とする
- 失敗時の応答はトレースバックを直接表示せず、ユーザ向け文面で扱える内容とする

## 9. 未決事項・確認事項
- .i の関数抽出ルール（マクロ/インライン/条件分岐の扱い）
- 完了メールの送信方式（SMTPサーバ設定、再送方針）
- result.csv の列定義の最終確定
- エラーコード体系（例: INPUT_PATH_NOT_FOUND, INVALID_EMAIL）の最終確定

## 10. 後工程（基本設計）への引継ぎ情報
- 画面要件: 入力2項目（フォルダパス/メール）と確認ダイアログ（OK/キャンセル、キャンセル時入力保持）
- 画面構成要件: PHPによるHTML生成を前提とし、CSS/JavaScriptで視認性・操作性を確保する
- バックエンド要件: iファイル探索、関数抽出、正規化、一致比較、database.json日次更新
- 出力配置要件: result.csv と集約コピー先フォルダをユーザ指定パスと同一階層へ配置し、同名衝突時は連番付きフォルダで回避する
- 異常時要件: 標準エラー応答（status=error, error_code, message, run_id）で返却し、画面はユーザ向け文面で表示
- 運用要件: 問題発生時はイシュー登録し、原因工程から再実施
- 差分開発情報: v140→v150でWindows環境のフォルダパス空白対応要件が追加されたため、入力検証・実行導線・受入観点を更新する

## 12. 差分開発情報（v140 → v150）
### 12.1 変更一覧
- 追加: Windows環境でフォルダ名に空白を含むフォルダパス入力を受け付ける要件
- 変更: なし
- 削除: なし
- 影響なし: 入力項目、確認ダイアログ、iファイル抽出・比較、database.json日次更新、出力配置、デプロイ条件

### 12.2 影響範囲
- 基本設計: Windowsパス入力仕様（空白許容）と入力検証方針を追記
- 詳細設計/実装: フロントエンドからバックエンド呼び出し時に空白を含むパスが欠損・分割されないようにする
- 評価: 空白を含むフォルダパスで正常実行できる観点を追加

### 12.3 回帰確認観点
- 入力～確認ダイアログ～送信の既存導線が維持されること
- キャンセル時の入力保持動作が維持されること
- パス不正/メール不正時のエラー表示が維持されること
- 既存のバックエンド処理結果（result.csv生成・集約コピー・メール通知）に影響がないこと
- Windows環境で空白を含むフォルダパス指定時も、通常パスと同様に処理できること

## 2. システム概要
本システムは、フロントエンド（PHP）とバックエンド（Python）で構成されるWebアプリである。
ユーザが指定したパスの .i ファイル群を解析し、社内サーバ保管済み .i 情報と関数単位で比較して再利用可否を判定する。

## 3. システム構成
### 3.1 論理構成
- フロントエンド: 入力受付、確認ダイアログ、実行要求送信、PHPによるHTML生成とCSS/JavaScriptによる画面表現
- バックエンド: iファイル探索、抽出、正規化、比較、出力生成、通知
- データ層: 社内サーバ情報JSON、database.json、実行時テンポラリJSON

### 3.2 配置構成
- Webサーバ: XAMPP Apache
- サーバ種別: オンプレミス（Windows Server 2019）
- デプロイ先: C:\xampp\htdocs\github\test1

## 4. データフロー
1. ユーザがフォルダパスとメールアドレスを入力
	- Windows環境では空白を含むフォルダパスを許容し、入力値を保持したまま後続処理へ受け渡す
2. 入力値不正時はフロントエンドが入力エラー画面を表示して処理を停止
3. 入力値妥当時は確認ダイアログで入力内容確認
4. OKでバックエンドへ実行要求
5. バックエンドが入力値と i ファイルの文字コードを判定（UTF-8 / Shift-JIS）し、UTF-8へ正規化
6. バックエンドが i ファイル探索・関数抽出・正規化を実施
7. database.json（社内サーバ情報）と関数単位で比較
8. result.csv と集約コピーを生成（ユーザ指定パスと同一階層へ配置、同名衝突時は連番付き新規フォルダを生成して上書き回避）
9. 完了メール送信

## 5. 機能構成（コンポーネント）
| コンポーネント | 主責務 | 入力 | 出力 |
|---|---|---|---|
| UI（PHP + HTML/CSS/JavaScript） | パラメータ受付、確認表示、要求送信、視認性を考慮した画面表示 | パス、メール | 実行要求 |
| Request Handler（Python） | 要求受付、処理制御、文字コード受理 | 実行要求 | 実行結果 |
| Encoding Normalizer | 文字コード差異吸収（UTF-8/Shift-JIS→UTF-8） | 入力値、iファイル | UTF-8正規化データ |
| IFile Analyzer | 探索、抽出、正規化 | ユーザ指定パス | 抽出JSON |
| Matcher | 関数単位一致判定 | 抽出JSON、database.json | 判定結果 |
| Exporter | CSV生成、ファイル集約、同名フォルダ衝突回避（連番付与） | 判定結果 | result.csv、成果物フォルダ（ユーザ指定パス同一階層） |
| Notifier | 完了通知 | 実行結果、メールアドレス | メール送信 |
| DB Updater | 社内サーバ情報更新 | 社内サーバ情報 | database.json |
| Error Formatter | 例外の標準化・画面表示用変換 | 例外情報、実行ID | 標準エラー応答 |

## 6. データ設計（基本）
- database.json: 社内サーバ情報 + iファイル情報
- temporary_*.json: 実行時抽出データ
- 社内サーバJSON: 評価結果サーバの原情報（追加/変更/削除あり）

## 7. インターフェース方針
- FE→BE: 実行要求（入力パス、メールアドレス、UTF-8/Shift-JIS受理）
- BE内部I/F: 解析結果JSONを介した疎結合連携
- BE→利用者: result.csv、集約成果物（同名衝突時は連番付きフォルダ）、メール通知
- BE→FE（異常時）: 標準エラー応答（error_code, message, run_id）
- FE表示: 標準エラー応答をユーザ向けメッセージへ整形して表示
- Windowsパス方針: 空白を含むフォルダパスを1つの文字列として保持し、分割・欠損なくBEへ受け渡す

### 7.0 文字コード方針
- 画面入力および i ファイル入力は UTF-8 と Shift-JIS（Windows-31J/CP932）を受け付ける。
- バックエンド内部処理および JSON/CSV 出力は UTF-8 に統一する。
- 文字コード不正時は標準エラー応答（`status=error`）で返却する。

### 7.1 応答JSON要約
- 成功時（status=ok）
	- `{"status":"ok","run_id":"...","matched_count":<number>,"total_count":<number>,"result_csv":"...","copied_dir":"..."}`
- 失敗時（status=error）
	- `{"status":"error","error_code":"<code>","message":"<text>","run_id":"..."}`

### 7.2 応答JSON項目定義
#### 成功応答（status=ok）
| 項目 | 必須 | 型 | 説明 | 制約 |
|---|---|---|---|---|
| status | 必須 | string | 実行結果ステータス | `ok` 固定 |
| run_id | 必須 | string | 実行識別子 | `YYYYMMDD_HHMMSS_ffffff` 形式 |
| input_path | 必須 | string | 入力フォルダパス | 空文字不可 |
| matched_count | 必須 | number | 一致件数 | `0` 以上 |
| total_count | 必須 | number | 判定対象総件数 | `0` 以上 |
| result_csv | 必須 | string | 結果CSVの出力パス | 実行環境の絶対パス |
| copied_dir | 必須 | string | 一致結果コピー先フォルダ | 実行環境の絶対パス |

#### 失敗応答（status=error）
| 項目 | 必須 | 型 | 説明 | 制約 |
|---|---|---|---|---|
| status | 必須 | string | 実行結果ステータス | `error` 固定 |
| error_code | 必須 | string | エラー分類コード | `INPUT_PATH_NOT_FOUND` など |
| message | 必須 | string | ユーザ表示可能なエラー説明 | 空文字不可 |
| run_id | 必須 | string | 実行識別子 | `YYYYMMDD_HHMMSS_ffffff` 形式 |
| detail | 任意 | string | 補足情報 | 未設定可 |

### 7.3 error_code定義（基本設計）
| error_code | 発生条件 | FE表示方針 |
|---|---|---|
| INPUT_PATH_NOT_FOUND | 指定入力パスが存在しない | パス見直しを促す |
| INVALID_EMAIL | メール形式が不正 | メール形式修正を促す |
| I_FILE_NOT_FOUND | 入力配下に `.i` が存在しない | 対象フォルダ確認を促す |
| INVALID_INPUT | その他入力不正 | 入力項目見直しを促す |
| INTERNAL_ERROR | 想定外エラー | 実行ID付きで再試行/連絡を促す |

### 7.4 FE判定ルール
- `status=ok` の場合は完了表示を行う。
- `status=error` の場合は `message`、`error_code`、`run_id` を表示する。
- JSON解析不能または必須項目欠落の場合は `INTERNAL_ERROR` 相当として扱う。
- トレースバック全文は通常表示せず、詳細表示領域に限定する。

### 8.3 画面遷移とUI責務
| 画面/状態 | 遷移条件 | UI責務 | 次状態 |
|---|---|---|---|
| 入力フォーム | 初期アクセス | フォルダパス/メール入力を受け付ける | 入力待ち |
| 入力フォーム | 決定押下かつ入力妥当 | 確認ダイアログへ入力値を表示する | 確認ダイアログ |

| 結果表示 | 処理成功/失敗 | 成功情報または標準エラー情報を表示する | 入力フォーム（再実行） |

## 9. 例外時方針
- 異常検知時は処理停止
- イシュー登録をトリガ
- 原因工程へ差戻し
- 異常応答はトレースバック非表示とし、標準エラー形式で返却する

## 12. 要件トレーサビリティ（要約）
- 要件入力（フォルダパス/メール）→ UI（PHP）入力フォーム + Request Handler（Python）入力I/F
- 要件入力（Windows空白パス許容）→ UI（PHP）入力保持 + FE→BE受け渡しI/F（単一文字列維持）
- 要件入力不正時の画面表示（フォルダパス/メール）→ UI（PHP）入力バリデーション + FE表示方針
- 要件画面構成（PHPでHTML生成 + CSS/JavaScriptで見やすい表示）→ UI（PHP + HTML/CSS/JavaScript）
- 要件入力文字コード互換（UTF-8/Shift-JIS）→ Request Handler（Python）+ Encoding Normalizer
- 要件処理（探索/抽出/正規化/比較）→ IFile Analyzer + Matcher
- 要件出力（result.csv/集約コピー同一階層配置、同名衝突時は連番フォルダで上書き回避）→ Exporter
- 要件通知（完了メール）→ Notifier
- 要件異常応答（標準JSON + 画面向け表示）→ Error Formatter + FE表示方針
- 要件保守（利用手順更新先を HOWTOUSE.md とする）→ 運用文書更新フロー（詳細設計で手順化）

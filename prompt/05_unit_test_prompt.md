# 05 単体評価プロンプト

## 目的
詳細設計通りに実装されているかを、正常系・異常系の両面で評価し、関数ごとにMC/DC（Modified Condition/Decision Coverage）100%達成を確認する。

## 入力
- `project/document/03_****.md`
- `project/src/frontend/`
- `project/src/backend/`

## 指示
1. Python側・JavaScript側それぞれのモジュール単位でテストケースを作成する。
   - Python: ツリー取得・読み込み（文字コード自動判定）・UTF-8保存・新規作成・削除・名前変更・バリデーション・`QWebChannel`メソッド
   - JavaScript: CodeMirror エディタ操作・marked.js プレビュー・Ctrl+S／保存ボタンイベント
2. 正常系と異常系を最低1件以上ずつ定義する（異常系: パス不正・ファイル未存在・文字コード読み取り失敗・保存失敗）。
3. 条件分岐を持つ各関数について、各基本条件が判定結果へ独立に影響することを示すMC/DCケース対を作成し、関数単位でMC/DC達成率100%を確認する。
4. 期待値・実績・判定（OK/NG）を記録する。
5. NG時は原因工程（03 or 04）を推定し記録する。
6. 結合評価向けに未解決課題を整理する。

## 出力形式
- 評価記録（Markdown/CSV）を `project/src/test/` に保存
- 関数別MC/DC達成結果（対象関数、判定条件、達成率、未達理由）を評価記録に含める

## 停止条件
- 仕様不足で評価不能な場合は停止し、必要情報を明記する。

## 工程ゲート（次工程進行確認）
- 評価成果物生成後、06工程へ進む前にGitHub使用者へチャットベースのレビューを実施する。
- チャットでは変更前の状態、変更後の状態と変更の考え方、懸念事項・特記事項（ある場合）、判断を仰ぎたい質問（ある場合）を説明する。
- GitHub使用者が承認した場合のみ06工程へ進む。未承認の場合は05工程で修正し再確認を実施する。
